სესია 5: გაფართოებული სავარჯიშოები
Support Triage და KYC Onboarding Flow-ები

19 პრაქტიკული დავალება Make.com-ის გაფართოებული მოდულების (Iterators, Aggregators, Error Handling), ავტომატური კატეგორიზაციის პრომპტების და მერჩანტების KYC Onboarding ფლოუს დასანერგად.

მოდული I: Process Redesign (AS-IS → TO-BE)
🔄 AS-IS Decomposition
1

არსებული პროცესის დეკომპოზიცია (AS-IS)

აირჩიე შენი დეპარტამენტის ერთი განმეორებადი პროცესი (მაგ. ბილინგის რეპორტის მომზადება, ტექნიკოსების გადანაწილება, ან მერჩანტების შეთანხმება). აღწერე მისი მიმდინარე ნაბიჯები, დახარჯული დრო, ადამიანური რესურსი და მთავარი "ტკივილის წერტილები" (Pain Points).

🔄 TO-BE Architecture
2

ახალი სქემის დაგეგმვა (TO-BE)

გარდაქმენი EX 1-ში აღწერილი პროცესი TO-BE სქემად: სად ჩაერთვება AI და სად დარჩება ადამიანის პასუხისმგებლობა (Human-in-the-loop)? ჩამოაყალიბე ნაბიჯები.

🔄 Prioritization Matrix
3

პრიორიტეტიზაცია: Impact × Complexity

შეაფასე შენი დაგეგმილი TO-BE პროცესი 1-დან 10-მდე შკალით: 1. ბიზნეს გავლენა (Impact) და 2. დანერგვის სირთულე (Complexity). გამოიყენე ფორმულა Priority Score = (Impact * 2) - Complexity და ახსენი მიღებული შედეგი.

მოდული II: Support Triage და ავტომატური რუტინგი
📥 Categorization Prompt
4

ბილეთის კატეგორიზაციის პრომპტის დაწერა

დავალება: ჩამოაყალიბე პრომპტი Claude-სთვის, რომელიც ბუნებრივ ქართულ ენაზე დაწერილ ტექნიკურ საჩივარს (მაგ. "აპარატმა 50 ლარიანი ჩაყლაპა და ქვითარი არ მომცა") ავტომატურად მიაკუთვნებს 4 ძირითად კატეგორიას (TECHNICAL, FINANCIAL, NETWORK, HARDWARE) და დააბრუნებს მხოლოდ JSON ობიექტს.

📥 Routing Rules
5

როუტერის ფილტრაციის წესები (Routing rules)

ჩამოაყალიბე ლოგიკური წესები Make.com-ის როუტერისთვის: რა პირობით უნდა გაიგზავნოს ბილეთი Slack-ის არხში `#ops-critical` (მაგ. თუ ლოკაციაა METRO ან BUS_STATION და პრიორიტეტია URGENT) და რა შემთხვევაში გადავიდეს ჩვეულებრივ საოპერაციო რიგებში.

📥 Slack Notification payload
6

Slack Alert-ის სტრუქტურირებული შეტყობინება

დაწერე Slack-ის შეტყობინების სტრუქტურა და ტექსტი (Markdown-ით) კრიტიკული ხარვეზის საჩვენებლად: უნდა ჩანდეს ტერმინალის ID, მისამართი, AI-ს მიერ შეჯამებული პრობლემა ქართულად და რეკომენდებული მოქმედება ტექნიკოსისთვის.

📥 Low Confidence Fallback
7

დაბალი ნდობის კოეფიციენტის (Low Confidence) დამუშავება

დაწერე ლოგიკა: როგორ უნდა მოვიქცეთ, თუ Claude API-ს მიერ დაბრუნებული `confidence_score` არის 0.6-ზე ნაკლები? როგორ გადავამისამართოთ იგი ადამიანთან (Human-in-the-loop) და რა სტატუსი მივანიჭოთ ტიკეტს?

მოდული III: Make.com Advanced & Error Handling
⚙️ Iterator / Aggregator
8

მასივების დამუშავება (Batch Processing)

აღწერე, როგორ გამოიყენებ Iterator-ს და Array Aggregator-ს, რათა Google Sheets-დან ამოღებული 20 გახსნილი ბილეთი გაატარო სათითაოდ Triage სცენარში, ხოლო ბოლოს გააერთიანო ერთ სუფთა ანგარიშად და საღამოს გაუგზავნო ხელმძღვანელს.

⚙️ Retry Handler
9

API Rate Limit (429) და Retry დირექტივა

როგორ დააკონფიგურირებ Make.com-ის Error Handler-ს (Retry მოდულით), თუ Claude API-დან დაბრუნდა HTTP 429 შეცდომა (მოთხოვნების ლიმიტის გადაჭარბება)? რამდენი მცდელობა და რა ინტერვალია ოპტიმალური?

⚙️ Resume Fallback
10

Resume დირექტივის გამოყენება

შექმენი სცენარი, სადაც Claude API-ის გათიშვისას (Timeout / 500 Error) ვიყენებთ Resume დირექტივას. რა სტრუქტურის Fallback მონაცემები უნდა გადაეცეს სცენარს (მაგ. category=UNKNOWN, priority=NORMAL), რათა შემდეგმა მოდულმა მუშაობა ჩვეულებრივად გააგრძელოს?

⚙️ Break directive
11

Break დირექტივა და გაჩერებული რიგები

როდის არის მიზანშეწონილი "Break" დირექტივის გამოყენება "Ignore"-ის ნაცვლად OPPA-ს ფინანსური ტრანზაქციების დალოგვისას?

მოდული IV: KYC Onboarding Flow
🔍 Webhook Trigger
12

B2B რეგისტრაციის ფორმის და Webhook-ის ინტეგრაცია

აღწერე Custom Webhook-ის შექმნის პროცესი Make.com-ზე, რომელიც მიიღებს მონაცემებს მერჩანტის რეგისტრაციის მომენტში (business_name, owner_name, id_number, business_type, location, application_date).

🔍 Document Review
13

ამონაწერის ვალიდაციის პრომპტი (OCR + AI)

დაწერე პრომპტი Claude-სთვის, რათა მან გადაამოწმოს ახალი B2B მერჩანტის მონაცემები: 1. არის თუ არა პირადი ნომერი 11 ციფრიანი, 2. არის თუ არა ბიზნეს ტიპი დასაშვები (GAMBLING/ADULT — არა, RETAIL/FOOD — კი), 3. კეთდება თუ არა ავტო-დასტური 80-ზე მაღალი ქულის შემთხვევაში.

🔍 Router Conditions
14

KYC-ის დამტკიცების სამმხრივი როუტერი

დაწერე კონკრეტული პირობები (Conditions) Make.com-ის როუტერის 3 გზისთვის: 1. APPROVED (Score >= 80, Low Risk), 2. MANUAL_REVIEW (Score 50-79, Medium Risk), 3. REJECTED (Score < 50, High Risk).

🔍 Sheets & Email Logging
15

სტატუსების დალოგვა და მერჩანტის შეტყობინება

ჩამოაყალიბე ავტომატური იმეილის შაბლონი (HTML ფორმატში), რომელიც გაეგზავნება მერჩანტს AI-ს მიერ "REJECTED" სტატუსის მინიჭების შემთხვევაში, მიზეზის (rejection_reason_ka) ავტომატური მითითებით.

მოდული V: Action Plan & ROI კალკულატორი
🚀 30-Day Triage Commit
16

საოპერაციო Triage-ის დანერგვის გეგმა

ჩამოაყალიბე 3-ნაბიჯიანი გეგმა: როგორ მოახდენ OPPA.GE-ის არსებული ტექნიკური მხარდაჭერის ბილეთების მიმართვას Make.com-ის ვებჰუკზე რეალურ სატესტო გარემოში.

🚀 Error Handling Audit
17

შენი გუნდის სცენარების აუდიტი

გადახედე შენს სამუშაო პროცესს. სად არის ყველაზე დიდი რისკი, რომ API-მ დააბრუნოს შეცდომა (მაგ. ნელი ინტერნეტი ტერმინალზე, ბილინგის სერვერის გადატვირთვა)? აღწერე რა ტიპის Error Handler-ს დაამატებ იქ.

⚡ ROI Calculation
18

საათობრივი ROI დაზოგვის გამოთვლა

თუ Triage-ის ავტომატიზაციამ დაზოგა მხარდაჭერის ჯგუფის 5 თანამშრომლის დღეში 2 საათი მუშაობა, ხოლო KYC ავტომატიზაციამ დაზოგა დღეში 3 საათი: გამოთვალე კვირაში (5 სამუშაო დღე) ჯამურად დაზოგილი საათები და მიღებული ფინანსური ეფექტი (თუ 1 საათის ღირებულებაა 15 GEL).

🌐 n8n Self-Hosted Setup
19

ლოკალური n8n გარემოს გამართვა (Docker / npm)

ნაწილი A: დაწერე ნაბიჯ-ნაბიჯ გზამკვლევი, თუ როგორ გაუშვებ n8n-ს OPPA-ს ლოკალურ სერვერზე Docker-ის მეშვეობით და როგორ დაუკავშირებ მას PostgreSQL ბაზას მონაცემების შესანახად.

ნაწილი B (Compliance & Privacy): ახსენი, რატომ აკმაყოფილებს ლოკალურად გაშვებული n8n ეროვნული ბანკის მოთხოვნებს PII (პერსონალური მონაცემების) დაცვის შესახებ, მაშინ როდესაც Make.com-ის SaaS ვერსია საჭიროებს დამატებით დაცვის ზომებს.

ნაწილი C (License Limitations): ჩამოთვალე n8n-ის Free Community ლიცენზიის 3 ძირითადი შეზღუდვა (მაგ. სკალირება, RBAC წვდომები, SSO) და ახსენი, რა საოპერაციო რისკებს ქმნის ეს ლიმიტები OPPA-ს მსხვილი წარმოების (Production) გარემოსთვის.
