სესია 5: გაფართოებული სავარჯიშოები
Support Triage და KYC Onboarding Flow-ები

15 პრაქტიკული დავალება Make.com-ის გაფართოებული მოდულების (Iterators, Aggregators, Error Handling), ავტომატური კატეგორიზაციის პრომპტების და მერჩანტების KYC Onboarding ფლოუს დასანერგად.

მოდული I: Support Triage და ავტომატური რუტინგი
📥 Categorization Prompt
1

ბილეთის კატეგორიზაციის პრომპტის დაწერა

დავალება: ჩამოაყალიბე პრომპტი Claude-სთვის, რომელიც ბუნებრივ ქართულ ენაზე დაწერილ ტექნიკურ საჩივარს (მაგ. "აპარატმა 50 ლარიანი ჩაყლაპა და ქვითარი არ მომცა") ავტომატურად მიაკუთვნებს 5-დან ერთ-ერთ კატეგორიას და დააბრუნებს მხოლოდ JSON ობიექტს.

📥 Routing Rules
2

როუტერის ფილტრაციის წესები (Routing rules)

ჩამოაყალიბე ლოგიკური წესები Make.com-ის როუტერისთვის: რა პირობით უნდა გაიგზავნოს ბილეთი Slack-ის არხში `#ops-critical` და რა შემთხვევაში ჩვეულებრივ საოპერაციო ბაზაში.

📥 Slack Notification payload
3

Slack Alert-ის სტრუქტურირებული შეტყობინება

დაწერე JSON Payload, რომელიც გაიგზავნება Slack Webhook-ში კრიტიკული ხარვეზის აღმოჩენისას, რათა ოპერატორმა ადვილად წაიკითხოს ტერმინალის ID, მისამართი და AI-ს მიერ შეჯამებული პრობლემა.

📥 Low Confidence Fallback
4

დაბალი ნდობის კოეფიციენტის (Low Confidence) დამუშავება

დაწერე ლოგიკა: როგორ უნდა მოვიქცეთ, თუ Claude API-ს მიერ დაბრუნებული `confidence_score` არის 0.6-ზე ნაკლები? როგორ გადავამისამართოთ იგი ადამიანთან (Human-in-the-loop)?

მოდული II: Make.com Advanced & Error Handling
⚙️ Iterator / Aggregator
5

მასივების დამუშავება (Batch Processing)

აღწერე, როგორ გამოიყენებ Iterator-ს და Array Aggregator-ს, რათა მიიღო 10 ბილეთისგან შემდგარი JSON მასივი, გაატარო თითოეული AI ფილტრში და ბოლოს გააერთიანო ერთ სუფთა ანგარიშად.

⚙️ Retry Handler
6

API Rate Limit (429) და Retry დირექტივა

როგორ დააკონფიგურირებ Make.com-ის Error Handler-ს (Retry მოდულით), თუ Claude API-დან დაბრუნდა HTTP 429 შეცდომა (მოთხოვნების ლიმიტის გადაჭარბება)? რამდენი მცდელობა და რა ინტერვალია ოპტიმალური?

⚙️ Resume Fallback
7

Resume დირექტივის გამოყენება

შექმენი სცენარი, სადაც API-ს გათიშვისას (Timeout) ვიყენებთ Resume დირექტივას. რა სტრუქტურის Fallback მონაცემები უნდა გადაეცეს სცენარს, რათა შემდეგმა მოდულმა მუშაობა ჩვეულებრივად გააგრძელოს?

⚙️ Break directive
8

Break დირექტივა და გაჩერებული რიგები

როდის არის მიზანშეწონილი "Break" დირექტივის გამოყენება "Ignore"-ის ნაცვლად OPPA-ს ფინანსური ტრანზაქციების დალოგვისას?

მოდული III: KYC Onboarding Flow
🔍 Webhook Trigger
9

B2B რეგისტრაციის ფორმის და Webhook-ის ინტეგრაცია

აღწერე Custom Webhook-ის შექმნის პროცესი Make.com-ზე, რომელიც მიიღებს მონაცემებს მერჩანტის რეგისტრაციის მომენტში (სახელი, საიდენტიფიკაციო კოდი, ატვირთული ამონაწერის ბმული).

🔍 Document Review
10

ამონაწერის ვალიდაციის პრომპტი (OCR + AI)

დაწერე პრომპტი, რომელიც დაავალებს AI-ს წაიკითხოს საჯარო რეესტრის ამონაწერიდან ამოღებული ტექსტი, შეამოწმოს რეგისტრაციის თარიღი (არის თუ არა 1 წელზე ნაკლები) და შეადაროს კომპანიის სტატუსი (აქტიური/შეჩერებული).

🔍 Router Conditions
11

KYC-ის დამტკიცების სამმხრივი როუტერი

დაწერე კონკრეტული პირობები (Conditions) Make.com-ის როუტერის 3 გზისთვის: 1. Auto-Approve (Risk: Low, match: true), 2. Manual-Review (Risk: Medium), 3. Reject (Risk: High ან match: false).

🔍 Sheets & Email Logging
12

სტატუსების დალოგვა და მერჩანტის შეტყობინება

ჩამოაყალიბე ავტომატური იმეილის შაბლონი (HTML ფორმატში), რომელიც გაეგზავნება მერჩანტს AI-ს მიერ "REJECT" სტატუსის მინიჭების შემთხვევაში, მიზეზის მითითებით.

მოდული IV: Action Plan და 30-დღიანი ვალდებულება
🚀 30-Day Triage Commit
13

საოპერაციო Triage-ის დანერგვის გეგმა

ჩამოაყალიბე 3-ნაბიჯიანი გეგმა: როგორ მოახდენ OPPA.GE-ის არსებული ტექნიკური მხარდაჭერის ბილეთების მიმართვას Make.com-ის ვებჰუკზე რეალურ გარემოში.

🚀 Error Handling Audit
14

შენი გუნდის სცენარების აუდიტი

გადახედე შენს სამუშაო პროცესს. სად არის ყველაზე დიდი რისკი, რომ API-მ დააბრუნოს შეცდომა (მაგ. ნელი ინტერნეტი ტერმინალზე, ბილინგის სერვერის გადატვირთვა)? აღწერე რა ტიპის Error Handler-ს დაამატებ იქ.

⚡ Complex Fallback Strategy
15

მრავალდონიანი Fallback-ის არქიტექტურა

დახატე ან აღწერე მრავალდონიანი Fallback სტრატეგია: თუ Claude 3.5 Sonnet გაითიშა, გადადის OpenRouter-ზე, თუ OpenRouter-იც გაითიშა — გადადის Gemini 1.5 Flash-ზე, ხოლო თუ სამივე გაითიშა — აგზავნის კრიტიკულ Alert-ს დეველოპერებთან და ინახავს ბილეთს ლოკალურად.
