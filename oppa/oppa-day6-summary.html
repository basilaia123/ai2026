2 Lovable

🧡 Lovable — OPPA-ს შიდა ინსტრუმენტების Prototyping

რა არის Lovable და რატომ ვიყენებთ მას?

Lovable.dev არის თანამედროვე AI Full-Stack App Builder, რომელიც იყენებს Supabase ბექენდს და React-ს. ის საშუალებას გვაძლევს მხოლოდ პრომპტების წერით შევქმნათ სრულფასოვანი შიდა პორტალები დეველოპერების გარეშე.

  • Terminal Status Monitor: OPPA-ს 8,000 ტერმინალის რეალურ დროში სტატუსების რუკა, ფილტრებით და შეცდომის კოდების დეტალებით.
  • Expense Approval App: შიდა საოპერაციო ხარჯების მოთხოვნის და დამტკიცების პორტალი, სადაც მენეჯერებს შეუძლიათ ღილაკზე ერთი დაჭერით დაამტკიცონ ან უარყონ მოთხოვნები.
  • Supabase Integration: მონაცემთა ბაზა და ავტორიზაცია იქმნება ავტომატურად.

3 SQL

📊 SQL ოპტიმიზაცია და Claude Code

ნელი Query-ებიდან მყისიერ პასუხამდე (45s → 0.3ms):

OPPA-ს ტრანზაქციების ბაზა შეიცავს მილიონობით ჩანაწერს. არასწორად დაწერილი მოთხოვნები იწვევს სერვერის გადატვირთვას. Claude Code-ის მეშვეობით ვახდენთ ბაზის ოპტიმიზაციას:

  • EXPLAIN ANALYZE: ავლენს Sequential Scan-ს (ბაზის მთლიანად გადაკითხვა) და გვიჩვენებს სად იხარჯება დრო.
  • CREATE INDEX CONCURRENTLY: ინდექსის შექმნა ლაივ ბაზაზე ისე, რომ ტრანზაქციების ჩაწერა არ შეფერხდეს.
  • Window Functions: საეჭვო AML პატერნების პოვნა (მაგალითად, 999 ლარიანი გადარიცხვები ბოლო 1 საათში) `PARTITION BY` და `OVER` გამოყენებით.

🔥 ნელი და ოპტიმიზებული AML SQL მოთხოვნის შედარება:

-- ნელი ვარიანტი (Sequential Scan 45 წამი): SELECT * FROM transactions t1 WHERE t1.amount >= 900 AND (SELECT COUNT(*) FROM transactions t2 WHERE t2.user_id = t1.user_id AND t2.created_at >= t1.created_at - INTERVAL '1 hour') >= 5; -- ოპტიმიზებული ვარიანტი Window Function-ით და ინდექსით (0.3 მილიწამი): -- ნაბიჯი 1: ინდექსის შექმნა CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_transactions_user_created ON transactions(user_id, created_at, amount); -- ნაბიჯი 2: ოპტიმალური Window query SELECT transaction_id, user_id, amount, created_at FROM ( SELECT transaction_id, user_id, amount, created_at, COUNT(*) OVER(PARTITION BY user_id ORDER BY created_at RANGE BETWEEN INTERVAL '1 hour' PRECEDING AND CURRENT ROW) as txn_count FROM transactions WHERE amount >= 900 ) sub WHERE txn_count >= 5;
4 Legacy Code

🛠️ Legacy კოდის მოდერნიზაცია და Security Fixes

Python 2 to Python 3 და უსაფრთხოების აუდიტი:

OPPA-ს ძველი სისტემები ხშირად შეიცავს 10+ წლის კოდს. Claude Code-ს ვიყენებთ ავტომატური რეფაქტორინგისთვის:

  • SQL Injection: ძველი სტრინგების კონკატენაციის ჩანაცვლება უსაფრთხო Parameterized SQL-ით.
  • Unit Testing: Pytest ტესტების ავტომატური გენერაცია კოდის სრული დაფარვისთვის (Coverage).
  • TypeScript Migration: JavaScript-ის ძველი ფაილების გადაყვანა ტიპიზებულ TypeScript-ში.
5 Hermes Agent

🧠 Hermes Agent — Autonomous AI და 4 OPPA Use Case

რა განასხვავებს Hermes Agent-ს Make.com-ისგან?

Hermes (Nous Research, 2026 წლის თებერვალი) არის **ავტონომიური AI Agent Framework**. Make.com-ისგან განსხვავებით, მას არ სჭირდება Flowchart-ის ხელით დახატვა. ის მუშაობს **Self-Improving Learning Loop**-ით: ყოველ წარმატებულ ამოცანას ინახავს Markdown "Skill" ფაილად ლოკალურ დისკზე (`~/.hermes/skills/`) და მომდევნო გაშვებაზე უფრო ეფექტურია.

Hermes Memory Layers:
- Working Memory (მიმდინარე სესია)
- Episodic Memory (უახლოესი წარსული ამოცანები)
- Semantic Memory (მუდმივი ფაქტები და OPPA-ს წესები)
- Skill Store (წარმატებული ამოცანებისგან მიღებული Markdown ინსტრუქციები)

🏦 OPPA-ს 4 კონკრეტული გამოყენების შემთხვევა:

  • AML მონიტორინგი (Autonomous): 24/7 ტრანზაქციების ანალიზი, საეჭვო პატერნების გამოკვლევა და ეროვნული ბანკისთვის STR ანგარიშის მომზადება.
  • Regulatory Reporting: კვარტალური ანგარიშგებების ავტომატური შევსება და ფორმატის ვალიდაცია.
  • DevOps ასისტენტი: სერვერების მონიტორინგი, ხარვეზისას Diagnostics სკრიპტების გაშვება და შეკეთების Skill-ის შექმნა.
  • Month-End Financial Close: ბილინგის და ბანკის მონაცემების მრავალდღიანი, რთული შედარება-შეჯერება (Multi-Day Workflow).

🔧 XML Tool Use & .env კონფიგურაცია:

Hermes 4 იყენებს XML სტრუქტურას ხელსაწყოების გამოსაძახებლად (JSON-ის ნაცვლად). კონფიგურაცია ხდება `.env` ფაილით OPPA-ს შიდა საოპერაციო სერვერზე (Self-hosted) მონაცემთა სრული უსაფრთხოებისთვის.

# .env კონფიგურაცია OPENROUTER_API_KEY=sk-or-v1-xxxx PRIMARY_MODEL=anthropic/claude-sonnet-4-5 FALLBACK_MODEL=nousresearch/hermes-4 SLACK_CHANNEL=#hermes-oppa SKILLS_DIR=~/.hermes/skills/oppa

🤖 Production AI ასისტენტი და Slack Bot

ტექნიკური მხარდაჭერის ბოტი OPPA API-ზე:

სესიის ფარგლებში ავაწყვეთ მოქმედი Slack Bot, რომელიც იკვებება OPPA-ს შიდა API დოკუმენტაციით და ეხმარება ინტეგრატორ მერჩანტებს ტექნიკური შეკითხვების პასუხში. ბოტის სისტემურ პრომპტში გაწერილია მკაცრი წესები (Guardrails):

  • მხოლოდ დოკუმენტაციის ფარგლებში: თუ კითხვა არ ეხება OPPA-ს API-ს, ბოტი თავაზიანად უარს ამბობს პასუხზე.

  • ლოკალური დაცვა: არ გადასცემს მერჩანტების API გასაღებებს მესამე მხარეს.
