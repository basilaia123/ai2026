⚙️ Make.com სცენარი — Support Triage Pipeline

ეს სცენარი ავტომატურად ამუშავებს Google Forms-ის (ან Webhook-ის) გზით შემოსულ Ticket-ებს:

1

Trigger: Google Forms / Webhook

ოპერატორი ავსებს ფორმას ან სისტემა Webhook-ს უგზავნის. Make.com იღებს: ticket_text, terminal_id, location_name, location_type.

2

Claude API: Triage Analysis

HTTP Module → api.anthropic.com. პრომპტი შეიცავს Ticket-ის მონაცემებს. Claude-ი აბრუნებს JSON: category, priority, department, slack_channel, summary_ka.

3

JSON Parse + Router

JSON Module ახდენს პასუხის პარსინგს. Router Module ანაწილებს ნაკადს 4 მიმართულებად: FIELD_TECH / ACCOUNTING / IT_OPS / MANAGEMENT.

4

Slack: არხის მიხედვით Alert

შესაბამის Slack-ის არხში იგზავნება: ტიკეტის №, ლოკაცია, პრიორიტეტი, summary_ka და suggested_action.

5

Google Sheets: ლოგი

ყველა Ticket ჩაიწერება Google Sheets-ში: timestamp, terminal_id, category, priority, status = "OPEN". KPI-ების გასათვლელად.

Advanced ლოგიკა — Multi-step Chains და Error Handling

Multi-step Prompt Chain — რა არის?

ერთი AI-ის მოთხოვნა ყოველთვის არ არის საკმარისი. კომპლექსური ამოცანები საჭიროებს თანმიმდევრულ ნაბიჯებს, სადაც ყოველი ნაბიჯი ეყრდნობა წინა ნაბიჯის შედეგს.

❌ Single-step (სუსტი)

„გააანალიზე ეს Ticket-ი, მოახდინე კლასიფიკაცია, გააგზავნე Slack-ში და ჩაწერე Sheets-ში.“

AI-ი ვერ ასრულებს ყველა ქმედებას. ვერ ახდენს ინტეგრაციას.

✅ Multi-step (სწორი)

ნაბიჯი 1: AI → JSON კლასიფიკაცია.
ნაბიჯი 2: Make.com პარსინგი.
ნაბიჯი 3: Slack/Sheets ინტეგრაცია.

Make.com არის „ოქროს კავშირი" AI-სა და სისტემებს შორის.

Escalation Chain — ავტომატური ესკალაცია

ნაბიჯი 2: Escalation Decision (პირველი AI-ის შედეგი გადაეცემა)

Support Ticket-ის Triage შედეგი:
- კატეგორია: {{triage_result.category}}
- პრიორიტეტი: {{triage_result.priority}}
- მდებარეობა: {{location_name}}
- სტატუსი: ტიკეტი 4 საათია გახსნილი, პასუხი არ ყოფილა.

გადაწყვიტე:
1. საჭიროა თუ არა ესკალაცია (escalation) მენეჯმენტზე? (true/false)
2. თუ true — შეადგინე ესკალაციის SMS/Email ტექსტი მენეჯერისთვის ქართულად.
3. თუ false — მომეცი შეხსენების (reminder) ტექსტი ტექნიკოსისთვის.

{
  "escalate": true | false,
  "message_type": "ESCALATION" | "REMINDER",
  "recipient": "manager@oppa.ge" | "tech-team@oppa.ge",
  "message_text": "..."
}

⚡ Error Handling — ფინანსური ავტომატიზაციის დაცვა

ავტომატიზაციაში შეცდომა ნიშნავს: დაკარგული ტიკეტი, გამოტოვებული ტრანზაქცია, SLA-ს დარღვევა. Fallback ლოგიკა სავალდებულოა.

🔴 გავრცელებული შეცდომები

  • ✗Claude API timeout (30 წამი+)
  • ✗AI-ი დააბრუნებს არავალიდურ JSON-ს
  • ✗Slack API rate limit
  • ✗Google Sheets row limit

🟢 Fallback სტრატეგია

  • ✓Retry Module (3 მცდელობა)
  • ✓JSON validation + Default ღირებულებები
  • ✓Error → #ops-critical Slack
  • ✓სარეზერვო Email-ის გაგზავნა ოპერატორთან

Make.com Error Handler — Fallback სქემა:

1. Action: Claude API → Triage JSON

2. Error? → Add Error Handler

├─ Log error → Google Sheets "errors" sheet

├─ Alert → Slack #ops-critical: "⚠️ AI Triage failed: {{error.message}}"

└─ Email → support@oppa.ge: "Manual review needed: Ticket {{terminal_id}}"

3. Resume → Default routing: category=UNKNOWN, priority=HIGH, department=IT_OPS

KYC Onboarding-ის ავტომატიზაცია — ლაივ Build

✅ ამ სეგმენტის მიზანი

ტრენერი Make.com-ში ლაივ-ში აშენებს OPPA-ს სავაჭრო ობიექტის (Merchant) Onboarding-ის სრული ავტომატიზაციის სცენარს. ახალი Merchant-ის რეგისტრაციიდან — Compliance გუნდის შეტყობინებამდე — ყველა ნაბიჯი ავტომატური იქნება.

KYC Onboarding — რა ხდება ამჟამად (ხელით)?

❌ ახლა (ხელით): 3-5 სამუშაო დღე

  1. Merchant ავსებს ქაღალდის ფორმას ან PDF-ს
  2. ოპერატორი ამოწმებს პირადობის დოკუმენტებს
  3. Compliance ოფიცერი ამოწმებს NAPR-ში
  4. Finance ამატებს CRM-ში
  5. Tech ასახავს ტერმინალის ID-ს

✅ AI-ით: 5-15 წუთი

  1. Webhook: Merchant ავსებს ონლაინ ფორმას
  2. Claude AI ამოწმებს დოკუმენტებს
  3. Router: Approved / Review / Rejected
  4. Slack: Compliance + Finance გუნდები
  5. CRM ავტო-განახლება (Zapier/API)

OPPA-ს KYC Onboarding — სრული Make.com სცენარი

1

Trigger: Webhook — New Merchant Registration

ონლაინ ფორმა (TypeForm/Google Forms) სარეგისტრაციო მონაცემებს Webhook-ით უგზავნის Make.com-ს: business_name, owner_name, id_number, business_type, location, docs_url.

2

HTTP Module: Claude API — KYC Document Check

Claude-ს ეგზავნება Merchant-ის მონაცემები (სახელი, ID ნომერი, ბიზნეს ტიპი, ლოკაცია). AI ამოწმებს სანდოობის ნიშნებს და OPPA-ს კრიტერიუმებს.

3

JSON Parse + Router: Verification Result

Claude-ის პასუხი პარსინგდება. Router-ი სამ გზად ყოფს: ✅ APPROVED → ავტო-Onboarding | ⚠️ MANUAL_REVIEW → Compliance Slack | ❌ REJECTED → Rejection Email.

4

Slack: Compliance + Finance გუნდების Alert

APPROVED: Slack #compliance-auto + Google Sheets. MANUAL_REVIEW: Slack #compliance-review + კომენტარი. REJECTED: ავტომატური rejection email Merchant-ს.

5

CRM / Google Sheets: Merchant ჩანაწერი

Google Sheets-ში ახალი რიგი: merchant_id (UUID), business_name, kyc_status, kyc_score, onboarding_date, assigned_terminal = TBD. Compliance-ს ეცნობება პრიორიტეტის მიხედვით.
