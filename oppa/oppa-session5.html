Vision AI დიაგნოსტიკა და Compliance სტატუსი

წინა სესიაში შევისწავლეთ PayBox-ის ეკრანების ფოტოებიდან Gemini Vision AI-ით შეცდომების ამოცნობა და Coda-სთან ინტეგრაცია. თუმცა, რეალურ ფინტექ პროცესებში გარე ღრუბლოვანი (Cloud) სერვისების გამოყენებისას ვეჯახებით ეროვნული ბანკის მკაცრ მოთხოვნებს:

📸 Gemini Vision AI-ის რეკაპი

Gemini 1.5-ის ინტეგრაცია Coda-ში (`=ASK_OPENAI`), სადაც აპარატის ეკრანის ფოტოდან ავტომატურად ხდება შეცდომის კოდის (მაგ. "Out of Paper") წაკითხვა და რეპორტინგი.

🔒 NBG Compliance (მონაცემთა დაცვა)

ეროვნული ბანკის წესებით, კლიენტის პერსონალური მონაცების გაგზავნა მესამე მხარის Cloud-ზე პირდაპირ იკრძალება. მონაცემები ან უნდა დაიმასკოს ლოკალურად გაგზავნამდე, ან სცენარი უნდა გაეშვას ლოკალურ, დაცულ სერვერზე (Self-hosted n8n).

Make.com საფუძვლები, Variable Mapping და ხელსაწყოების შედარება

📌 Make.com-ის ძირითადი ცნებები

Make.com არის ვიზუალური ავტომატიზაციის პლატფორმა, რომელიც აკავშირებს აპლიკაციებსა და API-ებს. იგი მუშაობს შემდეგი პრინციპით:

🔌

Trigger (ტრიგერი)

იწყებს სცენარს. არსებობს Polling (პერიოდული) და Instant (მყისიერი Webhook).

🧱

Module (მოდული)

სცენარის ცალკეული ნაბიჯი (მაგ. Google Sheets, Slack, HTTP).

⚡

Action (მოქმედება)

მოდული, რომელიც ასრულებს ქმედებას (იმეილის გაგზავნა, ბაზაში ჩაწერა).

🔀

Router (როუტერი)

ყოფს ნაკადს რამდენიმე ტოტად ფილტრების პირობების მიხედვით.

🔗 ცვლადების Mapping (Variable Mapping)

ავტომატიზაციაში უმნიშვნელოვანესია მონაცემების გადაცემა ერთი მოდულიდან მეორეში. Make.com იყენებს {{[ModuleID].[VariableName]}} სინტაქსს:

# ცვლადების დაკავშირების მაგალითი:

1. Google Form Trigger (Module ID: 1) ➔ აბრუნებს ველს: "Ticket Text"

2. Claude HTTP Action (Module ID: 3) ➔ იყენებს ცვლადს: {{1.ticket_text}}

3. Slack Post Action (Module ID: 5) ➔ პოსტავს Claude-ის პასუხს: {{3.body.choices[0].message.content}}

⏱️ სცენარების დაგეგმვა (Scheduling & Cron)

ყველა სცენარი არ უნდა მუშაობდეს მყისიერად. ზოგიერთი დავალება (მაგ. ბილინგის რეკონსილაცია) საჭიროებს დროის გრაფიკს:

⏱️ Regular Intervals

სცენარი ეშვება ყოველ X წუთში (მაგ. ყოველ 15 წუთში ერთხელ, რათა შეამოწმოს ოფლაინ ტერმინალები).

📅 Once a Day / Night Run

ეშვება კონკრეტულ საათზე (მაგ. ყოველ ღამე 02:00 საათზე, როდესაც ტერმინალების დატვირთვა მინიმალურია).

🗓️ Monthly Cron

ეშვება თვის კონკრეტულ რიცხვში (მაგ. ყოველი თვის 1 რიცხვში 09:00-ზე, ეროვნულ ბანკში რეპორტების გაგზავნამდე).

🌐 Webhook ტესტირების მეთოდოლოგია (Webhook.site & Postman)

სანამ სცენარს რეალურ ტრაფიკზე გავუშვებთ, საჭიროა ვებჰუკის სტრუქტურის შემოწმება და ტესტირება:

1. Webhook.site-ის გამოყენება

გამოიყენეთ Webhook.site დროებითი ტესტური URL-ის შესაქმნელად. გაგზავნეთ მასზე cURL მოთხოვნა, რათა დაინახოთ Payload-ის ზუსტი სტრუქტურა და headers-ი.

2. Make.com Webhook Capture

Make.com-ში Custom Webhook-ის შექმნისას დააჭირეთ "Redetermine data structure". გაგზავნეთ სატესტო რექვესთი Postman-იდან. Make ავტომატურად წაიკითხავს ველებს და შექმნის შესაბამის ცვლადებს.

🛠️ ალტერნატიული ავტომატიზაციის ხელსაწყოები: Zapier, Power Automate და Activepieces

მსხვილ კორპორაციებში Make-ისა და n8n-ის გარდა გამოიყენება სხვა პოპულარული პლატფორმებიც:

💻 Microsoft Power Automate (Desktop & Cloud)

ვისთვისაა: საუკეთესოა გუნდებისთვის, რომლებიც სრულად იყენებენ Microsoft 365-ს (Outlook, Excel, Teams).

🔥 მთავარი უპირატესობა: RPA (Robotic Process Automation) Desktop ვერსია, რომელიც ახდენს ძველი Windows პროგრამების ავტომატიზაციას კლიკების და კლავიატურის სიმულირებით.

⚡ Zapier (ყველაზე პოპულარული No-Code პლატფორმა)

ვისთვისაა: აბსოლუტური დამწყებებისთვის, ვისაც სურს 5 წუთში ააწყოს მარტივი ავტომატიზაცია.

🔥 მთავარი უპირატესობა: Zapier Central (AI Bots) — შეგიძლიათ plain English-ით აუხსნათ ბოტს, რა გააკეთოს და AI თავად ააწყობს ინტეგრაციას.

🟢 Activepieces (Open-Source Zapier-ის ლოკალური ალტერნატივა)

ვისთვისაა: კომპანიებისთვის, ვისაც სურს ლოკალურად გაშვებული (Self-hosted) მარტივი სისტემა.

🔥 მთავარი უპირატესობა: აქვს ისეთივე მარტივი და ლამაზი ინტერფეისი, როგორიც Zapier-ს, მაგრამ მისი კოდი ღიაა და n8n-ის მსგავსად შეგიძლიათ გაუშვათ OPPA-ს სერვერზე უფასოდ.

🔀 გადაწყვეტილების მიღების გზამკვლევი: Make vs n8n

როდის უნდა გამოვიყენოთ Make და როდის n8n? აქ მოცემულია მკაფიო შედარება OPPA-ს სპეციფიკის გათვალისწინებით:

ფაქტორი Make.com (გამოიყენე როცა...) n8n (გამოიყენე როცა...)
PII და NBG Compliance მონაცემები არ არის კრიტიკული ან სრულად დამასკულია. საჭიროა მონაცემების შენახვა ლოკალურად (On-Premise) OPPA-ს სერვერზე.
ინტეგრაციის სიჩქარე საჭიროა 10 წუთში აეწყოს Google Sheets-ის ან Slack-ის მზა მოდულები. საჭიროა შიდა სისტემებთან (API-ის გარეშე) ან SQL ბაზებთან პირდაპირი კავშირი.
კოდირება (Low-Code) გსურთ მარტივი ფორმულები და ლოგიკური Switch-ები. საჭიროა რთული JavaScript/Python კოდის გაწერა მონაცემთა მანიპულაციისთვის.

📊 ლიმიტები და ტარიფები (Free vs Core/Pro)

Make.com ტარიფები

  • Free Plan: 1,000 Operations თვეში, 100 MB Data transfer. 15-წუთიანი მინიმალური ინტერვალი.
  • Core/Pro: 10,000+ Operations, 1-წუთიანი მინიმალური ინტერვალი, Custom Webhooks, და როლების მართვა.

Anthropic (Claude) API ლიმიტები

  • Rate Limits: იზომება RPM (Requests Per Minute) და TPM (Tokens Per Minute).
  • Tier 1 (დამწყები): 50 RPM, 20,000 TPM.
  • Tier 4 (Enterprise): 4,000 RPM, 400,000 TPM.

Support Triage — Step-by-Step AI ავტომატიზაცია

⚙️ Triage სცენარის ნაბიჯ-ნაბიჯ აგება

ნაბიჯი 1: Webhook Intake და უსაფრთხოება

Make.com-ში ვქმნით Custom Webhook-ს. უსაფრთხოების დაცვა: ნებისმიერ გარე პირს, ვინც გაიგებს Webhook-ის URL-ს, შეუძლია მოთხოვნების გაგზავნა. უსაფრთხოებისათვის Make-ში ვამატებთ ფილტრს: headers-ში ვადარებთ **HMAC ხელმოწერას** (Signature Validation) ან ვფილტრავთ IP მისამართებს (IP Whitelisting), რათა რექვესთები მხოლოდ OPPA-ს ოფიციალური სერვერებიდან მივიღოთ.

🔒 ნაბიჯი 2: PII მონაცემების მასკირება (NBG Compliance)

სანამ Claude-ს გავუგზავნით ტექსტს, მასში უნდა წავშალოთ/დავფაროთ პერსონალური მონაცემები. ამისთვის Make-ში ვიყენებთ Text Parser-ს ან Formula-ს:

# ქართული პირადი ნომრის (11 ციფრი) მასკირება:

replace(1.ticket_text; "/[0-9]{11}/"; "XXXXXXXXXXX")

# ქართული მობილურის ნომრის (9 ციფრი) მასკირება:

replace(1.ticket_text; "/(5)[0-9]{8}/"; "5XXXXXXXX")

ნაბიჯი 3: Claude API Integration (კლასიფიკაცია)

გამოვიყენებთ Make-ის Anthropic მოდულს. ვუგზავნით დამასკულ ტექსტს და ვთხოვთ JSON კატეგორიზაციას (TECHNICAL, FINANCIAL, NETWORK).

💡 Claude Structured Output (Tool Use): პროდაქშენში უბრალო პრომპტით JSON-ის თხოვნამ შეიძლება შეცდომები გამოიწვიოს. გამოიყენეთ Claude-ის tools პარამეტრი (Structured Outputs). სქემის წინასწარ განსაზღვრით API გარანტირებულად დაგიბრუნებთ ზუსტ JSON სტრუქტურას, რომელსაც Make უშეცდომოდ წაიკითხავს.

🚨 ნაბიჯი 4: Escalation Chain (ავტომატური ესკალაცია)

თუ Claude-ის კლასიფიკაცია აჩვენებს, რომ ტრანზაქციის თანხა აღემატება 100 ლარს ან ობიექტი არის კრიტიკული, Claude-ს ენიჭება პრიორიტეტი `URGENT` და სცენარში ირთვება ავტომატური ესკალაციის ტოტი მენეჯერისთვის SMS შეტყობინების გასაგზავნად.

ნაბიჯი 5: Slack/Email Routing

Make.com-ში მასივების (Arrays) დასამუშავებლად გამოიყენება ორი ძირითადი მოდული:

Input: [ {id: 1}, {id: 2}, {id: 3} ]
➔ Output 1: {id: 1} (Bundle 1)
➔ Output 2: {id: 2} (Bundle 2)
➔ Output 3: {id: 3} (Bundle 3)

📊 2. Aggregator (ბანდლებიდან ➔ ერთ მასივად)

იღებს ბევრ ცალკეულ ოპერაციას და აერთიანებს მასივად.

Input 1: {id: 1}, Input 2: {id: 2}, Input 3: {id: 3}
➔ Output: [ {id: 1}, {id: 2}, {id: 3} ] (1 Single Array)
