1. Router & Conditions — პირობითი ლოგიკა

Router = "if/else" Make.com-ში. ერთი trigger-ი → სხვადასხვა მოქმედება პირობის მიხედვით.

მაგალითი — OPPA-ს კლიენტის მოთხოვნა:

Trigger: ახალი ფორმა შემოვიდა

Router:

├─ IF type = "complaint" → Slack #support-urgent + მენეჯერი

├─ IF type = "question" → AI პასუხი + Email

└─ IF type = "onboarding" → Onboarding Scenario

Make.com-ში: Add module → Router → ყოველ გზაზე Filter → condition-ის დაყენება. Filter-ი გამოიყენე "AND" / "OR" ლოგიკით.

2. Iterator — სიის დამუშავება

Iterator = ყოველ ელემენტს ერთი-ერთზე ამუშავებს. 50 კლიენტი → 50 ცალ-ცალკე ელ-წერილი.

Iterator სცენარი — OPPA-ს Weekly Digest

Make.com სცენარი:
1. Scheduled: ყოველ პარასკევს 17:00
2. Google Sheets: წაიკითხე "KPI" tab-ი — ყველა დეპარტამენტი
3. Iterator: ყოველ დეპარტამენტზე ცალ-ცალკე:
   a. ChatGPT/Claude: შეაჯამე ამ დეპარტამენტის KPI
   b. Slack: გაუგზავნე #dept-name არხში
   c. Google Sheets: ჩაწერე "sent ✅" სტატუსი
4. Aggregator: სულ გაგზავნილი — Slack #management

3. Error Handling — Fallback ლოგიკა

Production სცენარში API-ები ვარდება, მონაცემი არ მოდის. Error handler = სცენარი არ ჩერდება.

Break

სცენარი ჩერდება, შეცდომა ლოგდება

Ignore

შეცდომა იგნორდება, სცენარი გრძელდება

Retry

X-ჯერ ცდა, შემდეგ fallback მოდულზე

OPPA-ს სტანდარტი: ნებისმიერ API module-ზე → მარჯვენა კლიკი → "Add error handler" → Slack-ში შეტყობინება ადმინს.

4. Webhooks & HTTP — ნებისმიერ API-სთან

Webhook = Make.com-ი "ელოდება" გარე სიგნალს. HTTP module = Make.com-ი იძახებს ნებისმიერ API-ს.

Webhook — OPPA-ს გამოყენება

→CRM-ი ახალ კლიენტს ამატებს → webhook → Onboarding scenario იწყება
→გადახდის სისტემა → webhook → ანგარიში + Slack alert
→Typeform submission → webhook → Make.com → N სცენარი

HTTP Module — Custom API

→Claude API → AI პასუხი სცენარის შიგნით
→ეროვნული ბანკის კურსი → ავტო-განახლება
→ნებისმიერი REST API — Make.com-ს connector არ სჭირდება
