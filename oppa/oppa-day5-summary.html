  • Coda.io-ს საფუძვლები — AI სვეტები (`=ASK_OPENAI`) და მონაცემთა ინტეგრაცია GA4-თან და Google Ads-თან ცოცხალი ROAS-ის სათვლელად.
  • Advanced Data Analysis — ChatGPT-ის Python Sandbox-ის გამოყენება დიდი ფინანსური ანგარიშგებებისთვის და Outliers/ანომალიების აღმოჩენა ტრანზაქციებში.

2 Triage

📥 Support Triage — ავტომატური კლასიფიკაცია და რუტინგი

რატომ გვჭირდება AI Triage OPPA-ში?

OPPA-ს გააჩნია 8,000-ზე მეტი აქტიური ტერმინალი (PayBox), საიდანაც მხარდაჭერის გუნდში ყოველდღიურად ასობით ბილეთი (Ticket) შემოდის. AI-ს შეუძლია ბუნებრივ ენაზე დაწერილი ტექსტიდან მომენტალურად განსაზღვროს პრობლემის კატეგორია და პრიორიტეტი, რაც ამცირებს დამუშავების დროს 2 საათიდან 15 წამამდე.

  • კატეგორიზაცია: ბარათის ჩაყლაპვა, თანხის არასწორად ასახვა, ქაღალდის გაჭედვა, კომუნიკაციის გათიშვა.
  • Routing წესები: კრიტიკული ტერმინალები (მაგ. მეტროსადგურები, სავაჭრო ცენტრები) ხვდება პრიორიტეტულ `#ops-critical` Slack არხში, ჩვეულებრივი კი — სტანდარტულ CRM რიგში.
  • Structured Data: Claude API აბრუნებს სუფთა JSON ობიექტს Make.com-ის როუტერებისთვის.

🚀 პრომპტი Claude API-ით ბილეთების კლასიფიკაციისთვის:

შენ ხარ OPPA.GE-ს მხარდაჭერის ავტომატიზაციის ასისტენტი. გაანალიზე შემოსული ტექსტური ბილეთი და დააბრუნე პასუხი მხოლოდ JSON ფორმატში. კატეგორიები: - "CASH_ACCEPTOR_ISSUE" (თუ საუბარია ფულის გაჭედვაზე ან არმიღებაზე) - "PRINTER_ISSUE" (თუ საუბარია ქაღალდის დასრულებაზე ან გაჭედვაზე) - "TRANSACTION_NOT_REFLECTED" (თუ თანხა არ დაერიცხა კლიენტს) - "NETWORK_OFFLINE" (თუ კავშირის ხარვეზია) - "OTHER" (ყველა სხვა შემთხვევა) პრიორიტეტები: - "URGENT": თუ ტერმინალის ლოკაცია შეიცავს სიტყვას "METRO", "MALL", "AIRPORT" და თანხა აღემატება 100 ლარს. - "HIGH": თუ თანხა 100 ლარზე მეტია ან ბარათია ჩაყლაპული. - "MEDIUM": სხვა სტანდარტული შემთხვევები. JSON ფორმატი: { "category": "CASH_ACCEPTOR_ISSUE | PRINTER_ISSUE | TRANSACTION_NOT_REFLECTED | NETWORK_OFFLINE | OTHER", "priority": "URGENT | HIGH | MEDIUM", "confidence_score": 0.0-1.0, "summary_georgian": "მოკლე ქართული შეჯამება 1 წინადადებით", "needs_human_triage": true/false } ბილეთის მონაცემები: ლოკაცია: {{location}} ტექსტი: "{{ticket_text}}" თანხა: {{amount}} GEL
3 Error Handling

⚙️ Advanced Error Handling და API Fallbacks

Make.com-ის Error Handling დირექტივები:

როდესაც Make.com-ის API მოდული (მაგალითად, Claude / OpenRouter) აწყდება შეცდომას (Rate limit 429, Timeout ან Server Error 500), სცენარის გაჩერება არ შეიძლება. უნდა გამოვიყენოთ შემდეგი დირექტივები:

  • Retry (განმეორება): ხელახლა აგზავნის მოთხოვნას განსაზღვრული ინტერვალის შემდეგ (მაგ. 3-ჯერ, 5-წამიანი დაყოვნებით 429-ზე).
  • Resume (გაგრძელება): თუ მოთხოვნა ვერ სრულდება, აწვდის სცენარს წინასწარ გამზადებულ Mock/Fallback მონაცემებს და აგრძელებს მუშაობას.
  • Break (შეჩერება & შენახვა): ინახავს შეცდომას Make-ის Queue-ში და ელოდება ადმინისტრატორის მიერ ხელით გადაწყვეტას (Human-in-the-loop).
  • Ignore (იგნორირება): უგულებელყოფს შეცდომას და გადადის შემდეგ მოდულზე (მაგალითად, თუ Slack-ში შეტყობინება ვერ გაიგზავნა, მაგრამ ბილინგში მაინც უნდა დალოგდეს).

⚡ API Fallback-ის არქიტექტურა:

თუ Claude-ის ძირითადი API არხი წაიქცევა (Timeout / 500), Make.com-ის Router-ი ავტომატურად გადართავს მოთხოვნას **OpenRouter / Gemini Flash Fallback**-ზე, რითაც OPPA უზრუნველყოფს 99.9% საოპერაციო სტაბილურობას.

4 Make.com Advanced

⚡ Make.com-ის გაფართოებული ლოგიკა — Iterators, Aggregators და API ინტეგრაცია

მუშაობა მასივებთან (Batch Processing):

იმის ნაცვლად, რომ 100 ბილეთის დამუშავებაზე დავხარჯოთ 100 API ოპერაცია (რაც ძვირია და Rate Limit-ს გვახვედრებს), ვიყენებთ Batch მიდგომას:

  • Iterator: შლის JSON მასივს ცალკეულ ტრანზაქციებად ან ბილეთებად.
  • Router (ფილტრებით): ახდენს მონაცემების გაფილტვრას (მაგ. მხოლოდ 100 ლარზე მეტი თანხები გადაეცემა AI-ს).
  • Array Aggregator: აერთიანებს დამუშავებულ მონაცემებს კვლავ ერთ სუფთა მასივად, რათა ერთჯერადად გაიგზავნოს Slack-ში ან ჩაიწეროს Google Sheets-ში.
5 KYC Flow

🔍 KYC Onboarding — ლაივ სცენარი Make.com-ზე

ავტომატიზებული KYC Onboarding-ის სქემა:

როდესაც ახალი B2B მერჩანტი რეგისტრირდება OPPA-ს ქსელში, ხდება მისი დოკუმენტების ავტომატური შემოწმება (Verification Pipeline):

1. Webhook (რეგისტრაციის მოთხოვნა) -> 2. AI Check (დოკუმენტების ვალიდაცია: შეუსაბამობა, ვადები) -> 3. Router (ავტო-დამტკიცება / დამატებითი რევიუ / უარყოფა) -> 4. Notifications (Slack-ში ოპერატორს + Email-ი მერჩანტს) -> 5. Sheets (სტატუსის ჩაწერა)

🚀 KYC ვერიფიკაციის პრომპტი (OCR დამუშავება და AML შემოწმება):

შენ ხარ OPPA-ს KYC და AML შესაბამისობის ექსპერტი. გაანალიზე მერჩანტის სარეგისტრაციო დოკუმენტები და დააბრუნე მკაცრი JSON. მოცემული მონაცემები: - კომპანიის სახელი: {{company_name}} - საიდენტიფიკაციო კოდი: {{tax_id}} - დირექტორის სახელი: {{director_name}} - ამოღებული რეესტრის ტექსტი (OCR): "{{registry_extracted_text}}" დავალება: 1. შეამოწმე, ემთხვევა თუ არა რეესტრის ტექსტში მითითებული დირექტორი და საიდენტიფიკაციო კოდი შეყვანილ მონაცემებს. 2. შეამოწმე, არის თუ არა ამონაწერი მიმდინარე წლის (აქტუალური). 3. შეაფასე AML რისკი: კომპანიის საქმიანობა (მაგ. კრიპტოვალუტა, აზარტული თამაშები - მაღალი რისკია). JSON სტრუქტურა: { "data_match": true/false, "registry_valid": true/false, "aml_risk_score": "LOW | MEDIUM | HIGH", "reasoning_georgian": "დეტალური დასაბუთება ქართულად", "recommendation": "AUTO_APPROVE | MANUAL_REVIEW | REJECT" }
6 დანერგვა

🚀 30-დღიანი ვალდებულებების გეგმა

სესიის შემდეგ სავალდებულო ნაბიჯები დანერგვისთვის:

  • Support Triage Pipeline: ააწყვეთ Make.com-ზე 3-ეტაპიანი სცენარი შემოსული საცდელი ბილეთების კატეგორიზაციისა და Slack არხებში განაწილებისთვის.
  • API Fallback configuration: გაწერეთ OpenRouter / Gemini Flash როგორც Claude API-ის Fallback რესურსი თქვენს Make სცენარებში.
  • KYC Auto-routing test: შექმენით მერჩანტების რეგისტრაციის ფორმის სიმულაცია Google Forms + Webhook და გატესტეთ AI-ს მიერ "MANUAL_REVIEW"-ზე გაგზავნის სისწორე.
