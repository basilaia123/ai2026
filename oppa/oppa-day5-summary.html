სეგმენტი 3 · 6 OPPA ბიზნეს სცენარი

6 პოტენციური Make.com / n8n სცენარი OPPA-სთვის (Handout)

1. PayBox Kiosk ინკასაციის ოპტიმიზაცია

ბანკნოტის მიმღების 90% შევსებისას Webhook აჯგუფებს მარშრუტებს Google Maps API-ით და SMS-ით უგზავნის ინკასატორს.

2. ტრანზაქციების ყოველდღიური ავტომატური Reconciliation

ღამის 02:00 საათზე ბილინგის ბაზის და ბანკის ამონაწერების შედარება, შეუსაბამობისას Slack-არხში ტიკეტის შექმნა.

3. PayBox ქსელის მონიტორინგი და აღდგენის სკრიპტი (Auto-Reboot)

ტერმინალის ოფლაინში გასვლისას Reboot ბრძანების გაგზავნა GPRS როუტერზე და SLA ტიკეტის გახსნა Jira-ში.

4. მომხმარებელთა Viber/Telegram ჩატბოტი CRM სინქრონიზაციით

ჩარიცხვის პრობლემების ავტომატური გადამოწმება ბილინგში Claude API-ით და CRM-ში ჩაწერა.

5. ეროვნული ბანკის (NBG) Compliance ანგარიშების გენერატორი

n8n ლოკალური სერვერი აგროვებს მონაცემებს PEP პირებზე, აფილტრავს PII მონაცემებს და Compliance ოფიცერს უგზავნის ანგარიშს.

6. PayBox ინციდენტების ავტომატური ბაზა და Notion Knowledge Base Sync

ტექნიკური დაზიანებისას Claude API ადგენს Troubleshooting-ის გზას და ქმნის Notion Kanban ბარათს ტექნიკოსისთვის.

სეგმენტი 4 · Support Triage

Support Triage და PII მასკირება (NBG Compliance)

  • Webhook Security: HMAC ხელმოწერის ვალიდაცია და IP whitelisting.
  • PII მასკირების ფორმულა: {{replace(1.ticket_text; "/[0-9]{11}/"; "XXXXXXXXXXX")}} — პირადი ნომრებისა და მობილურის ნომრების წაშლა Claude API-ში გაგზავნამდე.
  • Escalation Chain: თუ ინციდენტის თანხა > 100 ლარია და კრიტიკულ მეტროს ლოკაციაზეა, Claude ანიჭებს პრიორიტეტს `URGENT` და სცენარში ირთვება მენეჯერის SMS ესკალაციის ტოტი.
  • Claude Structured Output (Tool Use): Claude API tools პარამეტრის გამოყენება მყარი JSON სტრუქტურის მისაღებად.
  • Error Handling: Claude API HTTP მოდულზე Error Handler-ის დამატება Resume (Fallback) და Ignore routes შეცდომების მართვისთვის.
  • Scenario Activation დისციპლინა: სცენარის გათიშვა (OFF) რედაქტირებისას, სატესტო გაშვება Run once-ით, და ჩართვა (ON) მხოლოდ შემოწმების შემდეგ.
  • ვერსიონირება & ტესტირება: სცენარის კლონირება [DEV] უსაფრთხო ტესტირებისთვის, მუშაობა ტესტურ Google Sheets/Slack არხში.
სეგმენტი 5 · KYC & Human-in-the-Loop

KYC Onboarding და Human-in-the-Loop (HITL) Workflow

  • AML Compliance Prompt: Claude-სთვის მკაფიო ცვლადების გადაცემა (`{{1.business_name}}`, `{{1.tax_id}}`) რისკის შეფასებისთვის.
  • Human-in-the-Loop (HITL): საეჭვო განაცხადები (`MANUAL_REVIEW`) იგზავნება Google Sheets-ში, სადაც ოპერატორის მიერ ჩექბოქსის ("Approved") მონიშვნა იწვევს მეორე სცენარის გააქტიურებას და ბილინგში გატარებას.
სეგმენტი 6 · ლუპები და ბატჩინგი

Loops, Iterators და Batch API ოპტიმიზაცია

  • Iterator / Aggregator: მასივების დაშლა და აწყობა მონაცემთა სტრუქტურების მართვისთვის.
  • Make Data Stores: შიდა Key-Value ბაზების გამოყენება ოფლაინ ტერმინალების ბოლო სტატუსის შესანახად და Alert Fatigue-ის პრევენციისთვის.
  • Batch Processing: 5 ტიკეტის გაერთიანება და Claude API-თან ერთი მოთხოვნით გაგზავნა (HTTP JSON payload configuration).
სეგმენტი 7 · მონიტორინგი და ხარჯები

მონიტორინგი, ხარჯები და რეალური ROI ანალიზი
