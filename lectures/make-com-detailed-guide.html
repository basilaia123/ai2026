რა არის Make.com?

Make (ყოფილი Integromat) არის მძლავრი ვიზუალური ავტომატიზაციის პლატფორმა, რომელიც მომხმარებლებს საშუალებას აძლევს, დააკავშირონ სხვადასხვა აპლიკაციები და სერვისები, რათა შექმნან კომპლექსური ავტომატური სამუშაო პროცესები (workflows) კოდის ერთი ხაზის დაწერის გარეშე. მისი მთავარი უპირატესობა ვიზუალური "drag-and-drop" ინტერფეისია, სადაც თითოეული ავტომატიზაცია დიაგრამის სახეს იღებს, რაც პროცესის გაგებასა და მართვას აადვილებს. Zapier-ისგან განსხვავებით, რომელიც ხაზობრივ ლოგიკას მიჰყვება, Make გაძლევთ საშუალებას, შექმნათ მრავალგანშტოებიანი, ციკლური და რთული სცენარები.

ისტორია და ევოლუცია

Make დაარსდა 2016 წელს Integromat-ის სახელით ჩეხეთში. 2022 წელს კომპანიამ გაიარა რებრენდინგი და გახდა Make.com. დღეს Make-ს იყენებს 300,000+ კომპანია და ინდივიდუალური მომხმარებელი მთელ მსოფლიოში, ხოლო პლატფორმა მხარს უჭერს 1,900+ აპლიკაციას.

💡 რჩევა ფასებთან დაკავშირებით

Operations vs Tasks: Make-ში ითვლება "ოპერაციები" (operations), Zapier-ში კი "ტასკები" (tasks). ერთი Zapier task შეიძლება შეესაბამებოდეს 3-5 Make operation-ს, რის გამოც Make ხშირად უფრო ხელმისაწვდომია.

დამატებითი Operations: თუ ლიმიტი ამოგეწურათ, შეგიძლიათ შეიძინოთ დამატებითი operations-ების პაკეტები:

  • 10,000 operations - $9
  • 100,000 operations - $79
  • 1,000,000 operations - $699

ძირითადი კონცეფციები დეტალურად

Scenarios (სცენარები)

სცენარი არის Make-ში ავტომატიზაციის სრული ლოგიკა, რომელიც შედგება მოდულებისგან. ესაა Zapier-ის "Zap"-ის ანალოგი, თუმცა ბევრად უფრო მოქნილი.
პროფესიონალური რჩევა: დაარქვით თქვენს სცენარებს აღწერითი სახელები და დაამატეთ ჩანაწერები (Notes) პირდაპირ ტილოზე, რათა დროთა განმავლობაში არ დაგავიწყდეთ, რას აკეთებს თითოეული ნაწილი.

Modules (მოდულები)

სცენარის შემადგენელი ბლოკები. მოდული შეიძლება იყოს ტრიგერი (რომელიც იწყებს სცენარს) ან მოქმედება (რომელიც ასრულებს დავალებას). ყოველი მოდულის გაშვება ითვლება ერთ "ოპერაციად", რაც თქვენი ტარიფის ლიმიტზე აისახება.
ხშირი შეცდომა: დამწყებები ხშირად ამატებენ ზედმეტ მოდულებს. ეცადეთ, ერთი მოდული გამოიყენოთ რამდენიმე მოქმედების შესასრულებლად, თუ მას ამის შესაძლებლობა აქვს (მაგ. 'Update Row' ნაცვლად 'Get Row' + 'Update Row').

Router (როუტერი)

ერთ-ერთი ყველაზე მძლავრი მოდული, რომელიც სცენარს ყოფს რამდენიმე განშტოებად, რაც პირობითი ლოგიკის (if/else) შექმნის საშუალებას იძლევა.
პროფესიონალური რჩევა: როუტერს აქვს "Fallback route" (სათადარიგო გზა), რომელიც მაშინ მუშაობს, თუ არცერთი სხვა ფილტრი არ დაკმაყოფილდა. გამოიყენეთ ის შეცდომების დასამუშავებლად ან დეფოლტ მოქმედების შესასრულებლად.

Iterator და Aggregator

Iterator იღებს მონაცემთა მასივს (მაგალითად, იმეილის რამდენიმე დანართს) და თითოეულ ელემენტს ცალ-ცალკე ამუშავებს. Aggregator კი პირიქით, რამდენიმე ცალკეულ ერთეულს ერთ მონაცემთა სტრუქტურად აერთიანებს (მაგ. ტექსტის ფაილის ან JSON-ის შესაქმნელად).
გასათვალისწინებელია: იტერატორი მოიხმარს იმდენ ოპერაციას, რამდენი ელემენტიც არის მასივში. 1000-ელემენტიანი მასივის დამუშავება 1000 ოპერაციას დაგიხარჯავთ.

Filters (ფილტრები)

ფილტრი არის პირობა, რომელიც განსაზღვრავს უნდა გაგრძელდეს თუ არა სცენარის შემდგომი ნაწილი. თუ პირობა არ სრულდება, სცენარი ჩერდება იმ route-ზე (ან fallback-ზე გადადის).
მაგალითი: გსურთ Slack შეტყობინება მხოლოდ იმ შემთხვევაში გაიგზავნოს, თუ Gmail-ის იმეილი შეიცავს სიტყვას "სასწრაფო". ფილტრი: Subject contains "სასწრაფო"

რჩევა: ფილტრები არ იხარჯავენ operations-ს! ისინი უფასოა და ეფექტური ინსტრუმენტია ოპერაციების დაზოგვისთვის.

Data Stores (მონაცემთა საცავი)

Make-ის შიდა მინი-ბაზა, რომელსაც შეუძლია მონაცემების შენახვა სცენარებს შორის. გამოიყენება:

  • State Management: რომელი იმეილები უკვე დამუშავდა (ID-ების ჩანაწერი)
  • Cache: ხშირად გამოყენებული მონაცემების დროებითი შენახვა
  • Counters: რაოდენობების თვლა (მაგ. რამდენი იმეილი გაგზავნილია დღეში)
  • Session Data: მომხმარებლის სესიის ინფორმაცია

შეზღუდვები: Data Store არ არის ნამდვილი database - მას აქვს ლიმიტები ჩანაწერების რაოდენობაზე და მოცულობაზე (Free: 100MB, Paid: 10GB).

Tools - დამხმარე მოდულები

Make-ის "Tools" კატეგორია შეიცავს უნივერსალურ მოდულებს მონაცემების დასამუშავებლად:

  • Set Variable / Get Variable: დროებითი ცვლადების შენახვა სცენარის შიგნით
  • Text Parser: ტექსტის დაყოფა, რეგექსით (regex) ინფორმაციის ამოღება
  • Sleep: სცენარის პაუზა გარკვეული დროით (1წმ-დან 300წმ-მდე)
  • Increment Function: რიცხვის გაზრდა/შემცირება
  • Compose a String: რთული სტრინგების შეკვრა

რჩევა: Text Parser-ით შეგიძლიათ ამოიღოთ ინფორმაცია ნებისმიერი ტექსტიდან regex-ის საშუალებით, რაც უფრო მოქნილია ვიდრე ჩაშენებული parsing ფუნქციები.

🚀 პირველი სცენარის შექმნა: ნაბიჯ-ნაბიჯ

მაგალითი: Gmail → Google Sheets ავტომატიზაცია

მიზანი: როდესაც ახალი იმეილი მოდის Gmail-ში კონკრეტული label-ით ("Leads"), მისი ინფორმაცია (გამგზავნი, თემა, თარიღი) ავტომატურად დაემატოს Google Sheets-ის ცხრილს.

ნაბიჯი 1: სცენარის შექმნა

  1. შედით Make.com-ზე და დააჭირეთ "Create a new scenario"
  2. გამოჩნდება ცარიელი canvas სადაც ააგებთ სცენარს

ნაბიჯი 2: ტრიგერის დამატება (Gmail)

  1. დააჭირეთ დიდ + ღილაკს canvas-ზე
  2. მოძებნეთ და აირჩიეთ Gmail აპლიკაცია
  3. აირჩიეთ ტრიგერი: "Watch emails"
  4. დააკავშირეთ თქვენი Gmail ანგარიში (OAuth)
  5. კონფიგურაცია:
    • Label: Leads
    • Limit: 1 (ყოველ გაშვებაზე 1 იმეილი)
    • Mark as read: Yes (რომ დუბლიკატები არ შეიქმნას)

ნაბიჯი 3: მოქმედების დამატება (Google Sheets)

  1. Gmail მოდულის შემდეგ დააჭირეთ +-ს
  2. მოძებნეთ Google Sheets
  3. აირჩიეთ "Add a row"
  4. დააკავშირეთ Google Sheets ანგარიში
  5. აირჩიეთ Spreadsheet და Sheet სადაც დაემატება ხაზი
  6. Mapping (მონაცემების დაკავშირება):
    • Column A (Sender): {{1.From Email}}
    • Column B (Subject): {{1.Subject}}
    • Column C (Date): {{1.Date}}
    • Column D (Body): {{1.Text}}

ნაბიჯი 4: ტესტირება

  1. დააჭირეთ "Run once" ღილაკს ქვედა მარცხენა კუთხეში
  2. გააგზავნეთ ტესტური იმეილი თქვენს Gmail-ზე "Leads" label-ით
  3. დაელოდეთ რამდენიმე წამს - სცენარი გაეშვება და დაამუშავებს იმეილს
  4. შეამოწმეთ Google Sheets - უნდა დაემატოს ახალი ხაზი

ნაბიჯი 5: აქტივაცია

  1. თუ ყველაფერი მუშაობს, დააჭირეთ "Scheduling" toggle-ს
  2. აირჩიეთ ინტერვალი (Free plan: 15 წუთი, Paid: 1-15 წუთი)
  3. გააქტიურეთ "ON"

  • HTTP Module: "უნივერსალური გასაღები" ნებისმიერ API-სთან დასაკავშირებლად, რომელსაც Make-ში ოფიციალური ინტეგრაცია არ აქვს.

20 იდეა სხვადასხვა სფეროსთვის

1. მარკეტინგი

ახალი ბლოგ პოსტის გამოქვეყნებისას, ავტომატურად შექმენით უნიკალური სურათი DALL-E-თი, დააპოსტეთ ყველა სოციალურ ქსელში და დაამატეთ Google Analytics-ში ანოტაცია.

2. გაყიდვები

როდესაც CRM-ში (მაგ. HubSpot) გარიგება "მოგებულად" აღინიშნება, ავტომატურად შექმენით კონტრაქტი Google Docs-ის შაბლონიდან და გაუგზავნეთ კლიენტს DocuSign-ით ხელმოსაწერად.

3. HR

ახალი თანამშრომლის დამატებისას (მაგ. BambooHR-ში), ავტომატურად შეუქმენით მას Google Workspace და Slack ანგარიშები და დაგეგმეთ მისი პირველი კვირის შეხვედრები კალენდარში.

4. ელ-კომერცია

როდესაც მომხმარებელი ტოვებს 5-ვარსკვლავიან შეფასებას, ავტომატურად დაგენერირეთ უნიკალური ფასდაკლების კოდი და გაუგზავნეთ მადლობის იმეილთან ერთად.

5. უძრავი ქონება

ყოველდღიურად შეამოწმეთ უძრავი ქონების საიტები HTTP მოდულით, გაფილტრეთ ახალი განცხადებები თქვენი კრიტერიუმებით (ფასი, უბანი) და გაუგზავნეთ დაიჯესტი პოტენციურ მყიდველებს.

6. განათლება

როდესაც სტუდენტი ავსებს Google Forms-ით შექმნილ ქვიზს, ავტომატურად შეაფასეთ, ჩაწერეთ ქულა Google Sheets-ში და გაუგზავნეთ სტუდენტს შედეგი პერსონალური უკუკავშირით.

7. ჯანდაცვა (ადმინისტრაცია)

დააავტომატეთ პაციენტებისთვის ვიზიტის შეხსენებების გაგზავნა SMS-ით (Twilio-ს გამოყენებით) და იმეილით ვიზიტამდე 24 საათით და 1 საათით ადრე.

8. პირადი პროდუქტიულობა

შექმენით დილის "ბრიფინგი", რომელიც ერთ იმეილში ან Slack შეტყობინებაში აერთიანებს დღის ამინდის პროგნოზს, მთავარ ამბებს RSS-დან და თქვენს კალენდარულ გეგმებს.

9. პროექტების მართვა

როდესაც Asana/Trello-ში იქმნება "სასწრაფო" პრიორიტეტის მქონე დავალება, ავტომატურად გაუგზავნეთ SMS შეტყობინება პროექტის მენეჯერს.

10. ფინანსები

გადაუღეთ ფოტო ქვითარს, ატვირთეთ Google Drive-ის კონკრეტულ ფოლდერში, გამოიყენეთ OCR მონაცემების ამოსაღებად და დაამატეთ ხარჯების აღრიცხვის ცხრილში.

11. იურიდიული სფერო

როდესაც CRM-ში ახალი პოტენციური პარტნიორი ემატება, ავტომატურად დააგენერირეთ NDA (კონფიდენციალურობის შეთანხმება) შაბლონიდან და გაუგზავნეთ მას განსახილველად.

12. არაკომერციული ორგანიზაციები

Stripe-ით ახალი დონაციის მიღებისას, ავტომატურად დაამატეთ დონორი Mailchimp-ის სიაში, გაუგზავნეთ პერსონალიზებული მადლობის იმეილი და აღრიცხეთ დონაცია მთავარ ცხრილში.

13. ფიტნეს ქოუჩინგი

კლიენტის გამოწერის დონის მიხედვით (რომელიც Airtable-შია შენახული), ავტომატურად გაუგზავნეთ მას ყოველდღიური სავარჯიშო გეგმები და სამოტივაციო ციტატები.

14. მუსიკა/პოდკასტები

როდესაც Dropbox-ის ფოლდერში ახალი აუდიო ფაილი აიტვირთება, ავტომატურად გააკეთეთ მისი ტრანსკრიფცია (მაგ. AssemblyAI-ს სერვისით) და შეინახეთ ტექსტი იმავე ფოლდერში.

15. მოგზაურობა

დააკვირდით ფრენის ფასებს კონკრეტულ მარშრუტზე API-ს საშუალებით. თუ ფასი გარკვეულ ზღვარს ჩამოსცდება, მიიღეთ Push შეტყობინება Pushover-ით ან Telegram-ით.

16. რესტორნები

მართეთ Google Form-ით შემოსული ჯავშნები: ავტომატურად დაამატეთ ისინი კალენდარში და გაუგზავნეთ მომხმარებელს დადასტურებისა და შეხსენების იმეილები.

17. გეიმინგ საზოგადოება

როდესაც Discord-ის სერვერს ახალი წევრი შემოუერთდება, ავტომატურად მიანიჭეთ მას "Newbie" როლი, გაუგზავნეთ მისასალმებელი წერილი წესებით და დაამატეთ "Welcome" არხზე.

18. მომხმარებელთა უკუკავშირი

ავტომატურად შეაგროვეთ უკუკავშირი სხვადასხვა წყაროდან (Twitter, იმეილები), გააანალიზეთ სენტიმენტი ChatGPT-ით და ნეგატიური შეფასებები გადაამისამართეთ პრიორიტეტულ არხზე Slack-ში.

19. კონტენტის კურატორობა

დააკვირდით კონკრეტულ RSS ფიდებს და Twitter-ის სიებს საკვანძო სიტყვებზე. რელევანტური სტატიის პოვნისას, ავტომატურად დაპოსტეთ ის გუნდის Slack-ის ან Discord-ის "კურირებული კონტენტის" არხში.

20. პირადი ფინანსები

დაუკავშირდით თქვენს საბანკო ანგარიშს (მაგ. Plaid-ის გავლით), მოახდინეთ ყოველდღიური ტრანზაქციების კატეგორიზაცია და შექმენით რეალური დროის ხარჯების დაშბორდი Google Sheets-ში.

Make.com vs Zapier: როდის რომელი?

კრიტერიუმი Make.com Zapier
ფასი უფრო ხელმისაწვდომი, მეტი ოპერაცია უფასო პაკეტში. იდეალურია მაღალი მოცულობის დავალებებისთვის. უფრო ძვირი, განსაკუთრებით რთული და მრავალსაფეხურიანი ავტომატიზაციისთვის.
ინტერფეისი ვიზუალური და მოქნილი, რაც საშუალებას გაძლევთ დაინახოთ მთლიანი პროცესი. მარტივი, მაგრამ ხაზობრივი და შეზღუდული. რთული ხდება კომპლექსური ლოგიკისას.
სირთულე შედარებით რთული შესასწავლია, მოითხოვს ტექნიკურ აზროვნებას. ძალიან მარტივია დამწყებთათვის, პირველი ავტომატიზაციის შექმნა წუთებშია შესაძლებელი.
ინტეგრაციები ნაკლები მზა ინტეგრაცია (1900+), თუმცა HTTP მოდულით ყველაფერთან დაკავშირება შეიძლება. გაცილებით მეტი მზა ინტეგრაცია (6000+).

მოწინავე ფუნქციები დეტალურად

HTTP/Webhooks: კარიბჭე ნებისმიერ სერვისთან

Make-ის HTTP მოდული მისი ყველაზე მძლავრი იარაღია. ის გაძლევთ საშუალებას, გააგზავნოთ მოთხოვნები (GET, POST, PUT, DELETE) ნებისმიერ ვებ-მისამართზე, რომელსაც აქვს API. Webhook-ი კი საშუალებას აძლევს გარე სერვისებს, რეალურ დროში "დაუძახონ" თქვენს Make სცენარს, როდესაც რაიმე მოვლენა ხდება.
განსხვავება: Polling (რასაც ბევრი ტრიგერი აკეთებს) ნიშნავს, რომ Make პერიოდულად ამოწმებს ცვლილებებს. Webhook-ი კი ნიშნავს, რომ ცვლილება თავად ატყობინებს Make-ს მომხდარის შესახებ, რაც უფრო სწრაფი და ეფექტურია.

HTTP Module-ის გამოყენების მაგალითი

Method: POST URL: https://api.openai.com/v1/chat/completions Headers: Authorization: Bearer YOUR_API_KEY Content-Type: application/json Body (JSON): { "model": "gpt-4", "messages": [{"role": "user", "content": "{{Gmail.Subject}}"}], "temperature": 0.7 }

ეს მაგალითი აგზავნის Gmail-ის თემას ChatGPT-ს და იღებს პასუხს, რომელსაც შემდეგ შეგიძლიათ გამოიყენოთ სხვა მოდულში.

შეცდომების მართვა (Error Handling): დაგეგმეთ წარუმატებლობა

პროფესიონალური ავტომატიზაცია ითვალისწინებს შეცდომებს. Make-ში შეგიძლიათ დაამატოთ Error Handler-ები, რომლებიც განსაზღვრავენ, რა მოხდეს, თუ მოდული შეცდომას დაუშვებს.

Error Handler-ის ტიპები:

  • Resume: გააგრძელე სცენარი, თითქოს შეცდომა არ მომხდარა
  • Break: შეაჩერე სცენარი და აღნიშნე როგორც წარმატებული
  • Rollback: დააბრუნე ყველა ცვლილება, რაც სცენარმა გააკეთა
  • Commit: შეინახე ყველა ცვლილება და შეაჩერე სცენარი
  • Ignore: უგულებელყოფა და გაგრძელება
მაგალითი: ... → Action: Send Email via Gmail   └─→ Error Handler (თუ მეილი ვერ გაიგზავნა):     ├─→ Directive: Resume (სცადე 3-ჯერ ყოველ 5 წუთში)     └─→ Action: Slack - Send message to #errors channel          ("მეილი ვერ გაიგზავნა: {{error.message}}")

✅ Best Practices - საუკეთესო პრაქტიკები

1. დასახელებები და ორგანიზაცია

  • აღწერითი სახელები: "Lead Capture: Form → CRM → Email" ნაცვლად "Scenario 1"
  • Folders: დააჯგუფეთ მსგავსი სცენარები ფოლდერებში (Marketing, Sales, HR)
  • Notes: დაამატეთ ჩანაწერები (Notes) მოდულებს რთული ლოგიკის ასახსნელად
  • Color Coding: გამოიყენეთ ფერები სხვადასხვა ტიპის route-ების გამოსაყოფად

2. ოპერაციების ოპტიმიზაცია

  • Filters-ის გამოყენება: ფილტრები უფასოა - გამოიყენეთ ისინი operations-ების დაზოგვისთვის
  • Batch Operations: მოძებნეთ მოდულები, რომლებსაც შეუძლიათ მრავალი ოპერაცია ერთად (მაგ. "Bulk update" ნაცვლად Iterator + Update)
  • Limit Results: ყოველთვის დააყენეთ Limit ტრიგერებზე (არ დაამუშავოთ 1000 ჩანაწერი ერთბაშად)
  • ავცილოთ დუბლიკატები: გამოიყენეთ Data Stores შემმოწმებელ ID-ებს, რომ ერთი ჩანაწერი ორჯერ არ დაამუშავოთ

3. უსაფრთხოება

  • API Keys: არასოდეს გააზიაროთ სკრინშოტები API გასაღებებით
  • Webhooks: გამოიყენეთ Webhook verification tokens
  • Sensitive Data: მგრძნობიარე მონაცემები არ შეინახოთ Data Store-ში უშიფრავად
  • Permissions: მიანიჭეთ აპლიკაციებს მხოლოდ საჭირო უფლებები (Principle of Least Privilege)

4. ტესტირება და მონიტორინგი

  • Test Data: ყოველთვის გატესტეთ რეალურ მონაცემებზე, არა მხოლოდ ფიქტიურზე
  • Run Once: გამოიყენეთ "Run once" ფუნქცია სანამ სცენარს გააქტიურებთ
  • Execution History: რეგულარულად გადახედეთ Execution History-ს შეცდომების დასამჩნევად
  • Notifications: დააყენეთ შეტყობინებები შეცდომებზე (Slack, Email)

5. დოკუმენტაცია

  • README Notes: შექმენით პირველი მოდული Text note-ით, რომელიც აღწერს სცენარის მიზანს
  • External Docs: შეინახეთ რთული სცენარების დოკუმენტაცია Notion-ში ან Google Docs-ში
  • Version Control: Make-ს არ აქვს ჩაშენებული version control - გააკეთეთ სცენარის JSON export backup-ებისთვის

❌ ხშირი შეცდომები და როგორ ავირიდოთ

1. ზედმეტი Operations-ები

შეცდომა: Iterator-ის გამოყენება 1000-ელემენტიან მასივზე → 1000 operations

გამოსწორება: გამოიყენეთ "Bulk" მოდულები, ან დაამუშავეთ batch-ებად (100-100 ელემენტი)

2. Error Handling-ის არარსებობა

შეცდომა: სცენარი ეშვება თუ ერთი API დაფეილდა

გამოსწორება: დაამატეთ Error Handler-ები კრიტიკულ მოდულებზე Resume ან Retry ლოგიკით

3. Rate Limits-ის იგნორირება

შეცდომა: სცენარი აგზავნის 1000 API request-ს წამში

გამოსწორება: გამოიყენეთ Sleep მოდული (მაგ. 100ms delay) ან გაიგეთ API-ს rate limits და დაიცავით ისინი

4. დუბლიკატების დამუშავება

შეცდომა: იგივე იმეილი 5-ჯერ დამუშავდა

გამოსწორება: გამოიყენეთ Data Store ID-ების შესანახად და შეამოწმეთ არსებობა სანამ დაამუშავებთ

5. ცუდი მონაცემთა Mapping

შეცდომა: {{1.email}} ნაცვლად {{1.from.email}} - შედეგად: undefined

გამოსწორება: გამოიყენეთ "Run once" და შეამოწმეთ რა სტრუქტურა აქვს თითოეული მოდულის output-ს

6. Webhook-ის Timeout

შეცდომა: Webhook-ის სცენარი 40 წამზე მეტს სრულდება და timeout-ს იღებს

გამოსწორება: გრძელი სცენარები დაყავით ორ ნაწილად: Webhook მიიღებს მონაცემს და ინახავს Data Store-ში, მეორე სცენარი (Scheduled) ამუშავებს მას

🧪 ტესტირება და დებაგინგი

1. Run Once - ყველაზე მნიშვნელოვანი ინსტრუმენტი

Run Once გამოიყენეთ სცენარის ნებისმიერი ცვლილების შემდეგ. ის აჩვენებს:

  • რა მონაცემები გადის თითოეულ მოდულს შორის (bubble view)
  • რამდენი operations დაიხარჯა
  • რომელ მოდულში მოხდა შეცდომა
  • რამდენ ხანში შესრულდა თითოეული მოდული

2. Data Inspector

დააჭირეთ რომელიმე მოდულის bubble-ს (თეთრ წრეწირს) Run Once-ის შემდეგ, რათა დაინახოთ:

  • Input: რა მონაცემები შემოვიდა მოდულში
  • Output: რა მონაცემები გამოვიდა
  • Metadata: დამატებითი ინფორმაცია (execution time, operations cost)

3. Incomplete Executions

თუ სცენარი შეიჩერა შეცდომის გამო და არ დაამუშავა ყველა მონაცემი, ის ინახება "Incomplete Executions"-ში. შეგიძლიათ:

  • გადახედოთ რა მოხდა
  • ხელით გაიმეოროთ (Resolve manually)
  • ავტომატურად გაიმეოროთ (Auto-resolve)

4. Execution History Filter

Execution History-ში შეგიძლიათ გაფილტროთ:

  • მხოლოდ შეცდომები (Status: Error)
  • კონკრეტული თარიღის range
  • კონკრეტული Data Bundle ID-თი

5. Debug-ის რჩევები

  • Text Note მოდული: დაამატეთ Text note მოდული workflow-ში log-ების დასაწერად
  • Slack Notifications: გაგზავნეთ შუალედური მონაცემები Slack-ში debug-ისთვის
  • Break Points: დაამატეთ "Break" მოდული რომ სცენარი შეჩერდეს კონკრეტულ წერტილში და დაინახოთ მონაცემები

🔥 სრული სცენარის მაგალითები

სცენარი 1: AI-Powered Customer Support Automation

მიზანი: ავტომატურად კატეგორიზება და პასუხი მომხმარებლის იმეილებზე AI-ის საშუალებით

Workflow: 1. Trigger: Gmail - Watch emails (Label: "Support") 2. Filter: Subject does NOT contain "Re:" (ახალი იმეილები) 3. HTTP: OpenAI API - Analyze sentiment & category - Prompt: "კატეგორიზაცია: Technical/Billing/General. Sentiment: Positive/Negative/Neutral" 4. Router: 3 routes based on category ├─→ Route 1 (Technical): │ ├─→ Airtable - Create record in "Tech Support Queue" │ └─→ Slack - Notify #tech-support channel ├─→ Route 2 (Billing): │ ├─→ Airtable - Create record in "Billing Queue" │ └─→ Slack - Notify #billing channel └─→ Route 3 (General): ├─→ HTTP: OpenAI - Generate auto-response └─→ Gmail - Send reply (auto-generated answer) 5. Final Action: Gmail - Add label "Processed"

Operations Count: 6-7 operations თითო იმეილზე

დროის დაზოგვა: ~10 წუთი თითო support ticket-ზე

სცენარი 2: Social Media Content Pipeline

მიზანი: ერთი კონტენტის იდეა → გაანალიზება AI-ით → მრავალ პლატფორმაზე გამოქვეყნება

Workflow: 1. Trigger: Google Sheets - Watch new rows (Sheet: "Content Ideas") 2. HTTP: OpenAI - Generate post variations - Input: {{Content Idea}} - Output: Twitter version (280 chars), LinkedIn version (long-form), Instagram caption 3. HTTP: DALL-E API - Generate image based on content 4. Google Drive: Upload generated image 5. Parallel Execution: (3 routes run simultaneously) ├─→ Twitter - Create tweet (with image) ├─→ LinkedIn - Create post (with image) └─→ Instagram - Schedule post via Buffer/Later API 6. Google Sheets: Update row - Mark as "Published" + add post URLs 7. Slack: Send summary to #marketing channel

Operations Count: 9-10 operations თითო content idea-ზე

დროის დაზოგვა: ~45 წუთი თითო post-ისთვის

სცენარი 3: Invoice Processing with OCR + AI

მიზანი: ინვოისის ფოტოს ან PDF-ის ავტომატური დამუშავება და ხარჯების აღრიცხვა

Workflow: 1. Trigger: Google Drive - Watch files in folder "Invoices/Inbox" 2. Google Cloud Vision: OCR - Extract text from image/PDF 3. HTTP: OpenAI API - Parse invoice data - Prompt: "Extract: Vendor Name, Amount, Date, Invoice Number, Category from OCR text" - Output: JSON structured data 4. Filter: Amount > 0 AND Date is valid (validation) 5. Google Sheets: Add row to "Expenses 2024" (all extracted data) 6. Data Store: Save invoice number (to avoid duplicates) 7. Router: Amount-based routing ├─→ Amount > $1000: │ └─→ Slack - Send approval request to CFO └─→ Amount <= $1000: └─→ Automatic approval 8. Google Drive: Move file to "Invoices/Processed" folder 9. Gmail: Send confirmation email to accounting team

Operations Count: 8-10 operations თითო ინვოისზე

დროის დაზოგვა: ~20 წუთი თითო ინვოისზე

სიზუსტე: 95%+ (AI-powered parsing)
